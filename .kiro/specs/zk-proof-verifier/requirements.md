# Requirements Document

## Introduction

This feature integrates a Groth16 ZK proof verifier into the Soroban smart contract layer on Stellar, enabling anonymous donations. Donors generate a zero-knowledge proof (Circuit 1) off-chain that proves they are authorised to donate a given amount without revealing their wallet address on-chain. The Soroban contract verifies the proof before accepting the donation, preserving donor privacy while maintaining on-chain auditability of the donation amount and fund allocation.

The existing donation flow (build → sign → submit) is extended: before the Stellar payment transaction is submitted, a ZK proof is generated client-side and verified by the Soroban contract. The 70/30 USDC split to the planting address and replanting buffer fund is preserved.

## Glossary

- **ZK_Proof**: A Groth16 zero-knowledge proof generated by the donor's client using Circuit 1.
- **Circuit_1**: The anonymous donation circuit that encodes the donation commitment without exposing the donor's wallet address.
- **Verifier_Contract**: The Soroban smart contract deployed on Stellar that verifies Groth16 proofs on-chain.
- **Proof_Input**: The public inputs to Circuit 1 — donation amount commitment and nullifier hash.
- **Nullifier**: A unique value derived from the donor's secret that prevents double-spending of the same proof.
- **Nullifier_Registry**: The on-chain set of spent nullifiers maintained by the Verifier_Contract.
- **Commitment**: A cryptographic hash of the donation amount and donor secret, used as a public input to the proof.
- **Proof_Generator**: The client-side library (snarkjs or equivalent) that generates ZK_Proofs from Circuit_1.
- **Verification_Key**: The Groth16 verification key corresponding to Circuit_1, embedded in the Verifier_Contract.
- **Anonymous_Donation_API**: The Next.js API route that orchestrates proof submission and Soroban contract invocation.
- **Donation_Service**: The existing `processDonationPayment` function in `lib/stellar/donation.ts`.
- **Planting_Address**: The Stellar account receiving 70% of each donation for direct tree planting.
- **Buffer_Address**: The Stellar account receiving 30% of each donation for the replanting buffer fund.

## Requirements

### Requirement 1: Client-Side ZK Proof Generation

**User Story:** As a donor, I want to generate a ZK proof on my device, so that my wallet address is never exposed on-chain during the donation.

#### Acceptance Criteria

1. WHEN a donor initiates an anonymous donation, THE Proof_Generator SHALL produce a valid Groth16 ZK_Proof using Circuit_1 inputs (donation amount commitment and nullifier).
2. WHEN the Proof_Generator completes, THE Proof_Generator SHALL return the ZK_Proof, the Proof_Input array, and the Nullifier to the calling code within 30 seconds.
3. IF the Proof_Generator fails to produce a proof, THEN THE Proof_Generator SHALL return a descriptive error message identifying the failure reason.
4. THE Proof_Generator SHALL derive the Nullifier deterministically from the donor's secret and a per-donation salt, such that the same secret and salt always produce the same Nullifier.
5. THE Proof_Generator SHALL format the ZK_Proof as a serialisable JSON object compatible with the Verifier_Contract's expected input schema.

---

### Requirement 2: On-Chain Groth16 Proof Verification

**User Story:** As the platform, I want the Soroban contract to verify ZK proofs on-chain, so that only valid anonymous donations are accepted without revealing donor identity.

#### Acceptance Criteria

1. WHEN a ZK_Proof and Proof_Input are submitted to the Verifier_Contract, THE Verifier_Contract SHALL verify the Groth16 proof against the embedded Verification_Key.
2. WHEN the Groth16 verification succeeds, THE Verifier_Contract SHALL return a success result to the caller.
3. IF the Groth16 verification fails, THEN THE Verifier_Contract SHALL return an error result with code `INVALID_PROOF`.
4. THE Verifier_Contract SHALL complete proof verification within a single Soroban transaction invocation.
5. THE Verifier_Contract SHALL expose a public `verify_proof` function accepting the ZK_Proof and Proof_Input as arguments.

---

### Requirement 3: Nullifier Double-Spend Prevention

**User Story:** As the platform, I want each ZK proof to be usable only once, so that a donor cannot replay the same proof to trigger multiple donations.

#### Acceptance Criteria

1. WHEN a ZK_Proof is verified successfully, THE Verifier_Contract SHALL check whether the Nullifier is present in the Nullifier_Registry.
2. IF the Nullifier is already present in the Nullifier_Registry, THEN THE Verifier_Contract SHALL return an error result with code `NULLIFIER_ALREADY_SPENT`.
3. WHEN a ZK_Proof passes verification and the Nullifier is not in the Nullifier_Registry, THE Verifier_Contract SHALL record the Nullifier in the Nullifier_Registry before returning success.
4. THE Nullifier_Registry SHALL persist Nullifier entries across Soroban contract invocations using Soroban persistent storage.
5. THE Verifier_Contract SHALL perform the nullifier check and registration atomically within the same contract invocation as proof verification.

---

### Requirement 4: Anonymous Donation API Endpoint

**User Story:** As the frontend, I want a single API endpoint to submit a ZK proof and trigger the donation, so that the anonymous donation flow is orchestrated server-side.

#### Acceptance Criteria

1. WHEN a POST request is received at `/api/transaction/build-anonymous-donation`, THE Anonymous_Donation_API SHALL accept a request body containing the ZK_Proof, Proof_Input, donation amount, and network identifier.
2. WHEN the request body is valid, THE Anonymous_Donation_API SHALL invoke the Verifier_Contract's `verify_proof` function via Soroban RPC.
3. WHEN the Verifier_Contract returns success, THE Anonymous_Donation_API SHALL build a Stellar payment transaction splitting the donation amount 70% to the Planting_Address and 30% to the Buffer_Address using USDC.
4. IF the Verifier_Contract returns `INVALID_PROOF`, THEN THE Anonymous_Donation_API SHALL return HTTP 422 with error code `INVALID_PROOF`.
5. IF the Verifier_Contract returns `NULLIFIER_ALREADY_SPENT`, THEN THE Anonymous_Donation_API SHALL return HTTP 409 with error code `NULLIFIER_ALREADY_SPENT`.
6. IF the request body is missing required fields, THEN THE Anonymous_Donation_API SHALL return HTTP 400 with a descriptive error message.
7. THE Anonymous_Donation_API SHALL not include the donor's wallet public key in the Soroban contract invocation arguments.

---

### Requirement 5: Anonymous Donation Transaction Signing and Submission

**User Story:** As a donor, I want to sign and submit the anonymous donation transaction with my wallet, so that the USDC transfer is authorised without linking my identity to the donation record.

#### Acceptance Criteria

1. WHEN the Anonymous_Donation_API returns a transaction XDR, THE Donation_Service SHALL present the transaction to the donor's connected wallet (Freighter or Albedo) for signing.
2. WHEN the donor signs the transaction, THE Donation_Service SHALL submit the signed transaction to the Stellar network.
3. WHEN the transaction is confirmed on the Stellar network, THE Donation_Service SHALL return the transaction hash to the caller.
4. IF the donor rejects the signing request, THEN THE Donation_Service SHALL return an error with message `USER_REJECTED_SIGNING`.
5. THE Donation_Service SHALL preserve the existing 70/30 USDC split (Planting_Address / Buffer_Address) for anonymous donations.

---

### Requirement 6: Verification Key Management

**User Story:** As a contract operator, I want the Groth16 verification key to be embedded in the Soroban contract at deployment, so that the key cannot be altered after deployment without redeploying the contract.

#### Acceptance Criteria

1. THE Verifier_Contract SHALL embed the Verification_Key for Circuit_1 as a compile-time constant within the contract WASM binary.
2. WHEN the Verifier_Contract is deployed, THE Verifier_Contract SHALL use the embedded Verification_Key for all subsequent `verify_proof` invocations.
3. THE Verifier_Contract SHALL expose a read-only `get_verification_key_hash` function that returns a SHA-256 hash of the embedded Verification_Key, enabling off-chain auditing.

---

### Requirement 7: Proof Serialisation Round-Trip

**User Story:** As a developer, I want ZK proof objects to serialise and deserialise correctly, so that proofs are not corrupted when transmitted between the client, API, and Soroban contract.

#### Acceptance Criteria

1. THE Proof_Generator SHALL serialise ZK_Proof objects to JSON strings.
2. WHEN a JSON string is deserialised back into a ZK_Proof object, THE Proof_Generator SHALL produce an object equal to the original ZK_Proof object.
3. FOR ALL valid ZK_Proof objects, serialising then deserialising SHALL produce an equivalent ZK_Proof object (round-trip property).
4. IF a JSON string does not conform to the ZK_Proof schema, THEN THE Proof_Generator SHALL return a descriptive parse error.

---

### Requirement 8: Network Support (Testnet and Mainnet)

**User Story:** As a developer, I want the ZK verifier to work on both Stellar testnet and mainnet, so that I can test the integration before deploying to production.

#### Acceptance Criteria

1. WHERE the network is `testnet`, THE Verifier_Contract SHALL be deployed to the Stellar testnet and the Anonymous_Donation_API SHALL invoke the testnet Soroban RPC endpoint.
2. WHERE the network is `mainnet`, THE Verifier_Contract SHALL be deployed to the Stellar mainnet and the Anonymous_Donation_API SHALL invoke the mainnet Soroban RPC endpoint.
3. THE Anonymous_Donation_API SHALL read the Verifier_Contract address for each network from environment variables `ZK_VERIFIER_CONTRACT_TESTNET` and `ZK_VERIFIER_CONTRACT_MAINNET`.
4. IF a network value other than `testnet` or `mainnet` is provided, THEN THE Anonymous_Donation_API SHALL return HTTP 400 with error message `UNSUPPORTED_NETWORK`.

# Requirements Document

## Introduction

This feature integrates Circuit 2 (location verification) into the Soroban smart contract layer on Stellar. Tree planters submit a zero-knowledge proof generated off-chain that proves their GPS coordinates fall within the approved Northern Nigeria geohash boundary, without revealing the exact coordinates on-chain. The Soroban contract verifies the location proof before recording a planting event, ensuring geographic eligibility while preserving planter location privacy.

This builds on the existing Groth16 ZK infrastructure established by Circuit 1 (anonymous donations). Circuit 2 uses the same Groth16 proving system and Soroban verification pattern, but encodes a geographic membership proof rather than a donation commitment.

## Glossary

- **Circuit_2**: The location verification ZK circuit that encodes proof of GPS coordinate membership within the approved geohash boundary without exposing exact coordinates.
- **Location_Proof**: A Groth16 zero-knowledge proof generated by the planter's client using Circuit_2.
- **Location_Proof_Input**: The public inputs to Circuit_2 — the geohash boundary commitment and a location nullifier.
- **Geohash_Boundary**: The approved Northern Nigeria geographic region encoded as a set of geohash cells at a fixed precision level, committed to as a Merkle root embedded in the Location_Verifier_Contract.
- **Boundary_Commitment**: A cryptographic Merkle root of the approved geohash cell set, embedded in the Location_Verifier_Contract at deployment time.
- **Location_Nullifier**: A unique value derived from the planter's secret and the planting event salt, preventing replay of the same location proof across multiple planting submissions.
- **Location_Nullifier_Registry**: The on-chain set of spent Location_Nullifiers maintained by the Location_Verifier_Contract.
- **Location_Proof_Generator**: The client-side library that generates Location_Proofs from Circuit_2 given GPS coordinates and the planter's secret.
- **Location_Verifier_Contract**: The Soroban smart contract that verifies Groth16 Location_Proofs on-chain using the embedded Boundary_Commitment and Verification_Key_2.
- **Verification_Key_2**: The Groth16 verification key corresponding to Circuit_2, embedded in the Location_Verifier_Contract.
- **Planting_API**: The Next.js API route that accepts a Location_Proof and orchestrates Soroban contract invocation and planting event recording.
- **Planting_Event**: An on-chain or off-chain record of a verified tree planting submission, including the location nullifier and timestamp.
- **GPS_Coordinates**: A latitude/longitude pair provided by the planter's device, used as private inputs to Circuit_2.
- **Geohash_Cell**: A rectangular geographic region encoded using the Geohash algorithm at a fixed precision (e.g., precision 4 ≈ 39 km × 20 km).

## Requirements

### Requirement 1: Client-Side Location Proof Generation

**User Story:** As a tree planter, I want to generate a ZK proof of my GPS location on my device, so that my exact coordinates are never transmitted or stored on-chain.

#### Acceptance Criteria

1. WHEN a planter initiates a planting submission with GPS_Coordinates, THE Location_Proof_Generator SHALL produce a valid Groth16 Location_Proof using Circuit_2 with the GPS_Coordinates as private inputs and the Boundary_Commitment as a public input.
2. WHEN the Location_Proof_Generator completes, THE Location_Proof_Generator SHALL return the Location_Proof, the Location_Proof_Input array, and the Location_Nullifier to the calling code within 60 seconds.
3. IF the GPS_Coordinates fall outside the approved Geohash_Boundary, THEN THE Location_Proof_Generator SHALL return an error with code `COORDINATES_OUT_OF_BOUNDARY` without generating a proof.
4. IF the Location_Proof_Generator fails to produce a proof for any other reason, THEN THE Location_Proof_Generator SHALL return a descriptive error message identifying the failure reason.
5. THE Location_Proof_Generator SHALL derive the Location_Nullifier deterministically from the planter's secret and a per-submission salt, such that the same secret and salt always produce the same Location_Nullifier.
6. THE Location_Proof_Generator SHALL format the Location_Proof as a serialisable JSON object compatible with the Location_Verifier_Contract's expected input schema.

---

### Requirement 2: On-Chain Groth16 Location Proof Verification

**User Story:** As the platform, I want the Soroban contract to verify location proofs on-chain, so that only planters within the approved Northern Nigeria region can submit planting events without revealing their exact GPS coordinates.

#### Acceptance Criteria

1. WHEN a Location_Proof and Location_Proof_Input are submitted to the Location_Verifier_Contract, THE Location_Verifier_Contract SHALL verify the Groth16 proof against the embedded Verification_Key_2.
2. WHEN the Groth16 verification succeeds, THE Location_Verifier_Contract SHALL confirm that the Location_Proof_Input includes the correct Boundary_Commitment matching the embedded value.
3. WHEN both the proof and boundary commitment checks pass, THE Location_Verifier_Contract SHALL return a success result to the caller.
4. IF the Groth16 verification fails, THEN THE Location_Verifier_Contract SHALL return an error result with code `INVALID_LOCATION_PROOF`.
5. IF the Boundary_Commitment in the Location_Proof_Input does not match the embedded Boundary_Commitment, THEN THE Location_Verifier_Contract SHALL return an error result with code `BOUNDARY_COMMITMENT_MISMATCH`.
6. THE Location_Verifier_Contract SHALL complete proof verification within a single Soroban transaction invocation.
7. THE Location_Verifier_Contract SHALL expose a public `verify_location_proof` function accepting the Location_Proof and Location_Proof_Input as arguments.

---

### Requirement 3: Location Nullifier Replay Prevention

**User Story:** As the platform, I want each location proof to be usable only once per planting submission, so that a planter cannot replay the same proof to register multiple planting events.

#### Acceptance Criteria

1. WHEN a Location_Proof is verified successfully, THE Location_Verifier_Contract SHALL check whether the Location_Nullifier is present in the Location_Nullifier_Registry.
2. IF the Location_Nullifier is already present in the Location_Nullifier_Registry, THEN THE Location_Verifier_Contract SHALL return an error result with code `LOCATION_NULLIFIER_ALREADY_SPENT`.
3. WHEN a Location_Proof passes verification and the Location_Nullifier is not in the Location_Nullifier_Registry, THE Location_Verifier_Contract SHALL record the Location_Nullifier in the Location_Nullifier_Registry before returning success.
4. THE Location_Nullifier_Registry SHALL persist Location_Nullifier entries across Soroban contract invocations using Soroban persistent storage.
5. THE Location_Verifier_Contract SHALL perform the nullifier check and registration atomically within the same contract invocation as proof verification.

---

### Requirement 4: Planting Submission API Endpoint

**User Story:** As the frontend, I want a single API endpoint to submit a location proof and register a planting event, so that the location-verified planting flow is orchestrated server-side.

#### Acceptance Criteria

1. WHEN a POST request is received at `/api/transaction/submit-planting`, THE Planting_API SHALL accept a request body containing the Location_Proof, Location_Proof_Input, planter identifier, and network identifier.
2. WHEN the request body is valid, THE Planting_API SHALL invoke the Location_Verifier_Contract's `verify_location_proof` function via Soroban RPC.
3. WHEN the Location_Verifier_Contract returns success, THE Planting_API SHALL record a Planting_Event containing the Location_Nullifier, planter identifier, and submission timestamp.
4. IF the Location_Verifier_Contract returns `INVALID_LOCATION_PROOF`, THEN THE Planting_API SHALL return HTTP 422 with error code `INVALID_LOCATION_PROOF`.
5. IF the Location_Verifier_Contract returns `BOUNDARY_COMMITMENT_MISMATCH`, THEN THE Planting_API SHALL return HTTP 422 with error code `BOUNDARY_COMMITMENT_MISMATCH`.
6. IF the Location_Verifier_Contract returns `LOCATION_NULLIFIER_ALREADY_SPENT`, THEN THE Planting_API SHALL return HTTP 409 with error code `LOCATION_NULLIFIER_ALREADY_SPENT`.
7. IF the request body is missing required fields, THEN THE Planting_API SHALL return HTTP 400 with a descriptive error message.
8. THE Planting_API SHALL not include the planter's GPS_Coordinates in the Soroban contract invocation arguments or in the Planting_Event record.

---

### Requirement 5: Boundary Commitment and Verification Key Management

**User Story:** As a contract operator, I want the Geohash boundary commitment and Groth16 verification key to be embedded in the Soroban contract at deployment, so that neither can be altered after deployment without redeploying the contract.

#### Acceptance Criteria

1. THE Location_Verifier_Contract SHALL embed the Verification_Key_2 for Circuit_2 as a compile-time constant within the contract WASM binary.
2. THE Location_Verifier_Contract SHALL embed the Boundary_Commitment (Merkle root of the approved Northern Nigeria geohash cell set) as a compile-time constant within the contract WASM binary.
3. WHEN the Location_Verifier_Contract is deployed, THE Location_Verifier_Contract SHALL use the embedded Verification_Key_2 and Boundary_Commitment for all subsequent `verify_location_proof` invocations.
4. THE Location_Verifier_Contract SHALL expose a read-only `get_verification_key_hash` function that returns a SHA-256 hash of the embedded Verification_Key_2, enabling off-chain auditing.
5. THE Location_Verifier_Contract SHALL expose a read-only `get_boundary_commitment` function that returns the embedded Boundary_Commitment, enabling off-chain verification that the correct geohash set was committed.

---

### Requirement 6: Location Proof Serialisation Round-Trip

**User Story:** As a developer, I want location proof objects to serialise and deserialise correctly, so that proofs are not corrupted when transmitted between the client, API, and Soroban contract.

#### Acceptance Criteria

1. THE Location_Proof_Generator SHALL serialise Location_Proof objects to JSON strings.
2. WHEN a JSON string is deserialised back into a Location_Proof object, THE Location_Proof_Generator SHALL produce an object equal to the original Location_Proof object.
3. FOR ALL valid Location_Proof objects, serialising then deserialising SHALL produce an equivalent Location_Proof object (round-trip property).
4. IF a JSON string does not conform to the Location_Proof schema, THEN THE Location_Proof_Generator SHALL return a descriptive parse error.

---

### Requirement 7: Geohash Boundary Encoding

**User Story:** As a developer, I want the approved Northern Nigeria region to be encoded as a deterministic set of geohash cells, so that the Boundary_Commitment can be reproduced off-chain for auditing and circuit compilation.

#### Acceptance Criteria

1. THE Location_Proof_Generator SHALL encode GPS_Coordinates as a Geohash_Cell at precision level 4 before evaluating boundary membership.
2. THE Location_Proof_Generator SHALL evaluate boundary membership by checking whether the encoded Geohash_Cell is a member of the approved geohash cell set.
3. THE Location_Proof_Generator SHALL compute the Boundary_Commitment as a Merkle root over the lexicographically sorted approved geohash cell set using SHA-256 as the hash function.
4. FOR ALL approved GPS_Coordinates, encoding to a Geohash_Cell and checking membership SHALL return true (membership correctness property).
5. FOR ALL GPS_Coordinates outside the approved region, encoding to a Geohash_Cell and checking membership SHALL return false (exclusion correctness property).

---

### Requirement 8: Network Support (Testnet and Mainnet)

**User Story:** As a developer, I want the location verifier to work on both Stellar testnet and mainnet, so that I can test the integration before deploying to production.

#### Acceptance Criteria

1. WHERE the network is `testnet`, THE Location_Verifier_Contract SHALL be deployed to the Stellar testnet and the Planting_API SHALL invoke the testnet Soroban RPC endpoint.
2. WHERE the network is `mainnet`, THE Location_Verifier_Contract SHALL be deployed to the Stellar mainnet and the Planting_API SHALL invoke the mainnet Soroban RPC endpoint.
3. THE Planting_API SHALL read the Location_Verifier_Contract address for each network from environment variables `ZK_LOCATION_VERIFIER_CONTRACT_TESTNET` and `ZK_LOCATION_VERIFIER_CONTRACT_MAINNET`.
4. IF a network value other than `testnet` or `mainnet` is provided, THEN THE Planting_API SHALL return HTTP 400 with error message `UNSUPPORTED_NETWORK`.
